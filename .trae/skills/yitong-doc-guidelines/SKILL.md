# 亿通文档编写规范

本技能提供亿通公司研发文档的编写规范和指南，包括5种标准文档类型。

## 文档类型概览

| 文档类型 | 命名规范 | 主要用途 |
|---------|---------|---------|
| 会议纪要 | YYYYMMDD会议纪要 | 记录会议内容和决议 |
| 软件设计说明书 | 软件设计说明书XXX | 详细设计文档 |
| 软件系统技术方案 | 软件系统技术方案模板 | 技术方案汇报 |
| 需求分析说明书 | 系统需求分析说明书XXX | 需求规格说明 |
| 需求变更文档 | YYYYMMDD-需求描述 | 需求变更记录 |

---

## 一、会议纪要

### 命名规范
- 标准格式：`YYYYMMDD会议纪要`
- 样例格式：`2025-05-09 海关监管处会议`

### 文档结构

| 字段 | 说明 |
|-----|------|
| 会议主题 | 会议的核心议题 |
| 会议时间 | 格式：YYYY-MM-DD HH:MM-HH:MM |
| 会议地点 | 会议召开地点 |
| 主持人 | 会议主持人 |
| 记录人 | 会议记录人员 |
| 参会人 | 参会人员名单 |
| 会议内容 | 分点记录会议讨论内容（1. 2. 3.） |
| 会议总结 | 会议决议和总结（1. 2.） |

### 编写要点
- 会议内容按条目分点记录
- 会议总结需明确决议事项
- 时间格式统一使用24小时制

## 二、软件设计说明书

### 命名规范
- 标准格式：`软件设计说明书XXX`
- 样例格式：`软件设计说明书XXX`

### 文档结构

#### 1. 引言
- **编写目的**：编写本设计说明书的目的
- **背景**：本项目的业务背景
- **定义**：关键字的解释
- **参考资料**：
  - 《YT-YF-01-001 软件产品设计开发控制程序（B0）》
  - 《需求分析方法指南》
  - 《软件开发流程填写规范》
  - 《技术安全管理规范》
  - 《单元测试方法指南》
  - 《命名规程》
  - 《软件详细设计说明书》
  - 《数据安全防护要求》
  - 《数据安全管理办法》

#### 2. 总体设计
- **需求概述**：系统整体需求描述+需求流程图
- **软件架构设计**：描述整体架构、层次结构、模块划分及关系
- **软件架构设计图**：架构图辅助说明
- **软件拓扑图**：系统拓扑结构图
- **模块功能概述**：各主要模块功能简要描述

#### 3. 数据库设计
- **数据库总体设计**：数据库类型、命名规范
- **数据库表设计**：
  - 表名
  - 表的描述
  - 字段名
  - 数据类型（int、varchar、datetime等）
  - 长度
  - 主键
  - 外键
  - 约束

#### 4. 系统详细设计
- **模块一**
  - 功能点
  - 功能详细说明
  - 输入
  - 输出
  - 限制条件
  - 处理逻辑
  - 接口设计
- **模块二**（同上结构）

#### 5. 接口设计
- **外部接口**：与外部用户软件系统的接口（Web服务、API等）
- **内部接口**：与公司内部其他软件系统的接口

#### 6. 系统出错处理设计
- **出错信息**：错误代码、错误描述、可能原因
- **补偿措施**：补救措施说明

#### 7. 非功能性设计
- **性能设计**：并发处理方案、缓存策略、数据库优化
- **可开行设计**：容灾备份、故障恢复机制
- **安全行设计**：数据加密、权限控制、日志审计
- **可维护性**：日志规范、监控告警、代码可读性

## 三、软件系统技术方案

### 文档信息
- 文档版本：V1.0
- 编制日期：YYYY年MM月DD日
- 编制单位：亿通/技术研发中心-数字口岸组

### 文档结构

#### 一、项目背景与目标
- **项目背景**：业务痛点、政策要求或市场需求（300-500字）
- **核心目标**：明确系统建设的核心价值（如"提升业务效率30%"）

#### 二、系统建设内容
- **建设内容**：项目建设内容说明+业务流程图
- **非建设范围**：明确暂不覆盖的内容（避免范围争议）

#### 三、技术方案核心思路（重点）

**（一）技术架构设计**
- 架构选型：微服务架构/云原生架构/B/S三层架构等
- 架构分层：前端交互层-API网关层-业务服务层-数据存储层
- 关键特性：弹性扩容、灾备能力等

**（二）核心技术栈选型**
- 分层技术清单：
  - 前端：Vue3 + Vite
  - 后端：Spring Cloud Alibaba
  - 数据库：DM + Redis
  - 部署：Docker + Kubernetes
- 选型依据：结合项目需求说明技术优势

**（三）关键技术亮点（汇报重点）**
- 差异化技术：解决项目核心痛点的特色技术
- 技术价值：技术对项目目标的支撑作用

**（四）技术可行性验证**
- 验证方式：压力测试、原型验证等
- 验证结论：技术方案可行性结论

#### 四、实施计划与里程碑

| 里程碑阶段 | 时间节点 | 核心交付物 |
|-----------|---------|-----------|
| 需求确认 | YYYY-MM | 需求说明书（简版） |
| 技术验证 | YYYY-MM | 技术原型demo + 验证报告 |
| 系统开发 | YYYY-MM | 核心功能开发完成 |
| 系统上线 | YYYY-MM | 可试用版本 + 上线方案 |

#### 五、风险与应对建议
- **潜在风险**：列举2-3个关键风险
- **应对方向**：针对性解决思路

#### 六、预期成效
- **业务成效**：如"降低人工操作成本20%"
- **管理价值**：如"实现业务数据可视化监控"

## 四、需求分析说明书

### 命名规范
- 标准格式：`系统需求分析说明书XXX`
- 样例格式：`XXX项目需求分析文档`

### 文档结构

#### 1. 引言
- **编写目的**：说明编写目的，指出预期读者
- **背景**：
  - 产品名称
  - 开发任务的提出者、开发者、用户
  - 本产品与其他产品的关系
- **定义**：专门术语的定义和缩写词原文
- **参考资料**：
  - 《YT-YF-01-001 软件产品设计开发控制程序（B0）》
  - 《需求分析方法指南》
  - 《软件开发流程填写规范》
  - 《软件配置规范》
  - 《技术安全管理规范》
  - 《命名规程》
  - 《项目计划书》
  - 《软件需求分析说明书》

#### 2. 需求概述
- **目标**：
  - 开发意图、应用目标及作用范围
  - 主要功能、处理流程、数据流程
  - 系统高层次图
- **运行环境**：硬件环境和支持环境
- **关键点**：关键功能、关键算法、关键技术
- **约束条件**：经费限制、开发期限、方法与技术等

#### 3. 需求规格
- **业务流程图**：总体功能/对象结构描述
- **描述约定**：通常使用的约定描述
- **功能或对象的描述**：
  - 功能或对象1
  - 说明
  - 功能点（新增、修改、删除、查询、下载等）
  - 接口（公司内部接口、外部接口）

#### 4. 其他需求
- **数据**：输入、输出数据及数据管理能力要求
- **安全**：访问控制、身份认证、数据加密、脱敏
- **性能**：响应时间、吞吐量、并发用户数、资源利用率
- **故障处理**：
  - 软件系统的问题
  - 错误信息
  - 补救措施

#### 5. 算法说明
- 每个主要算法的概况
- 每个主要算法的详细公式

#### 6. 尚未解决的问题
- 软件需求中的尚未解决的遗留问题

#### 7. 支持信息
- 附录、索引等

## 五、需求变更文档

### 命名规范
- 标准格式：`YYYYMMDD-需求描述`
- 样例格式：`20250331-新增抵离港时间校验/海关单据统计增加前十港/船舶分类`

### 文档结构

文档开头使用4行表格，格式如下：

| 项目名称 | (空) |
|---------|------|
| 变更提出方 | 监管单位/业务/企划/平台等 | 变更提出日期 | YYYY-MM-DD |
| 建议完成时间 | YYYY-MM-DD | 实际完成时间 | YYYY-MM-DD |
| 变更原因 | (空) |

**表格结构说明：**
- 第1行：项目名称 + 空单元格
- 第2行：变更提出方 + 内容 + 变更提出日期 + 日期
- 第3行：建议完成时间 + 日期 + 实际完成时间 + 日期
- 第4行：变更原因 + 空单元格

### 变更内容格式
```
变更1
描述具体变更内容，包括涉及哪些菜单、功能模块，及其影响范围。

变更2
同上
```

## 通用编写规范

### 文档命名规则
1. 日期格式统一使用YYYYMMDD
2. 文档名称简洁明了，体现文档类型和主题
3. 样例文档需标注"样例"字样

### 参考资料清单
所有研发文档可参考以下规范文件：
- 《YT-YF-01-001 软件产品设计开发控制程序（B0）》
- 《需求分析方法指南》
- 《软件开发流程填写规范》
- 《技术安全管理规范》
- 《单元测试方法指南》
- 《命名规程》
- 《软件详细设计说明书》
- 《数据安全防护要求》
- 《数据安全管理办法》
- 《软件配置规范》
- 《项目计划书》
- 《软件需求分析说明书》

### 编写建议
1. 技术方案文档部分内容可用AI生成
2. 技术亮点部分在汇报时可适当夸大
3. 实施计划与里程碑视汇报阶段决定是否包含
4. 所有文档需包含必要的图表辅助说明
